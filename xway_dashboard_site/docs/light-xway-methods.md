# Политика легких методов XWAY

## Цель

Обычные обновления не должны запускать старую тяжелую загрузку полного product payload. Сначала нужно использовать focused-методы, а кластерные/normquery-данные догружать только когда пользователь явно открывает глубокую вкладку или просит анализ фраз/кластеров.

## Тяжелый путь

Считать тяжелыми эти вызовы:

- `/api/products?campaign_mode=full`
- `/api/products?campaign_mode=summary&heavy_campaign_ids=...`
- кластерное расширение кампаний в `collectCampaignHeavyPayload`
- полную пагинацию статусов:
  - `campaignStatusMpHistoryFull`
  - `campaignStatusPauseHistoryFull`

Тяжелый путь допустим только для явных deep-сценариев: вкладки товара с кластерами, статусами или историей ставок, а также API-запросы с `include_product_heavy_details=true`.

## Легкие/focused методы

Перед любым тяжелым обновлением нужно использовать эти методы:

- `/api/ai/campaign-schedules` - настроенное время показов.
- `/api/ai/campaign-limits` - лимиты РК, потрачено, остаток лимита и правила бюджета.
- `/api/ai/campaign-budget-history` - пополнения бюджета и автопополнение.
- `/api/ai/campaign-bid-history` - история ставок на уровне кампаний.
- `/api/ai/campaign-status-history` - ограниченные логи статусов и пауз.
- `/api/catalog?aux=0` - строки каталога, остатки и базовое состояние товаров.
- `/api/catalog-product-details` с `campaign_details=1&best_time=0` - состояния РК, лимиты и расписания.
- `/api/catalog-product-details` с `campaign_details=0&best_time=1` - только лучшие часы заказов.
- `/api/catalog-chart` - абсолютные дневные метрики; `include_campaign_types=1` добавлять только когда нужна разбивка по типам РК.
- `/api/catalog-issues` - диагностика ошибок; внутри должен использоваться ограниченный `status-pause-history`, без полной пагинации.

## Текущие дефолты

- `/api/products` по умолчанию работает как `campaign_mode=summary`.
- Summary-режим больше не догружает первые кампании автоматически. Тяжелые детали грузятся только при наличии `heavy_campaign_ids`.
- Первый экран товара, фоновое обновление кеша, сравнение и предзагрузка графиков используют summary-режим.
- Глубокие вкладки товара лениво догружают детали только по видимым campaign id.
- AI recommendation data по умолчанию работает в summary; `detail_level=full` добавляет focused-методы по кампаниям, но не полный кластерный payload.
- `include_product_heavy_details=true` - явный escape hatch для старого тяжелого контекста товара.
- Pachka AI deep context использует focused-методы и обычные TTL source-cache по умолчанию. `PACHKA_REPORT_AI_AUTO_REFRESH=1` включать только если нужно принудительно обходить source-cache.
- Focused-истории кампаний и статусы кешируются на короткий TTL, чтобы один отчет не дергал XWAY повторно одними и теми же запросами.

## Чеклист аудита для будущих правок

1. Не добавлять новый вызов `/api/products?campaign_mode=full`, если он не стоит за явным deep-действием.
2. Не вызывать `campaignStatus*HistoryFull` в обновлении каталога, массовых обновлениях или scheduled jobs.
3. Для точечного вопроса предпочитать один focused endpoint вместо общего обновления товара.
4. Если кнопка обновляет много товаров, разбивать процесс на строки каталога, график, детали РК, лучшие часы и ошибки.
5. Если нужен кластерный анализ, запрашивать конкретную кампанию или кластер, а не все кампании по умолчанию.
