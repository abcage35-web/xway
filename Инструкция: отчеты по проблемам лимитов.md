# Инструкция по сборке отчетов XWAY: лимиты / бюджет

## Цель

Собирать Markdown-отчеты по лимитам и бюджету РК из XWAY DRR analytics.

Источник дашборда:

```text
https://xway-bt4.pages.dev/drr-analytics
```

Сейчас инструкция покрывает три отчета:

```text
Проблемы: настройка лимитов и бюджетов
Проблемы: вылеты лимитов
Проблемы: Автоисключения Поиска
```

Итоговые Markdown-файлы нужно отдавать пользователю в таком порядке:

```text
1. Проблемы: настройка лимитов и бюджетов.md
2. Проблемы: вылеты лимитов.md
3. Проблемы: Автоисключения Поиска.md
Инструкция: отчеты по проблемам лимитов.md
```

Для всех отчетов использовать один и тот же источник маркетологов из Google Sheet и один общий фильтр `Остаток FBO > 10`, если пользователь явно не попросил другое.

Отчет `Проблемы: настройка лимитов и бюджетов` должен содержать строки по РК, где не установлен лимит расхода или не настроено пополнение бюджета.

Отчет `Проблемы: вылеты лимитов` должен содержать строки по РК, которые в выбранном периоде простаивали из-за лимита расхода или нехватки бюджета не меньше установленного порога подряд.

Отчет `Проблемы: Автоисключения Поиска` должен содержать строки по CPM РК в статусе `ACTIVE` или `PAUSED`, где не настроено автоисключение поиска.

## Период

Использовать дефолт страницы: последние 3 дня до вчера.

Пример для запуска 2026-05-26:

```text
start = 2026-05-23
end   = 2026-05-25
```

Правило:

```text
end = вчера
start = end - 2 дня
```

## Формат итоговой таблицы

Колонки в Markdown-таблице:

```text
Артикул / XWAY / Название товара / Категория / Маркетолог / Остаток FBO / Тип РК / Проблема лимит расхода / Проблема пополнение бюджета
```

## Фильтры

В отчет попадают только строки, где выполнены все условия:

1. Товар есть в XWAY.
2. У товара есть РК в статусе `ACTIVE` или `PAUSED`.
3. По конкретному типу РК отсутствует лимит расхода или бюджет/пополнение.
4. `Остаток FBO > 10`.

Не включать РК со статусом `FROZEN`.

## Типы РК

Не выводить старые технические подписи:

```text
CPM Поиск
CPM Реком
Ед. CPM
CPC
```

Правильные подписи:

```text
unified       -> Единая ставка
manual_search -> Ручная ставка: поиск
manual_recom  -> Ручная ставка: рекомендации
cpc           -> Оплата за клики
```

Статус РК на момент сбора в `Тип РК` не добавлять. Нужен только бизнес-тип:

```text
Единая ставка
Ручная ставка: поиск
Ручная ставка: рекомендации
Ручная ставка: поиск + рекомендации
Оплата за клики
```

## Проверка проблем

Для каждой нормализованной РК со статусом `ACTIVE` или `PAUSED`.

Лимит расхода считается настроенным, если:

```js
Boolean(state.spend_limit_active) && Number(state.spend_limit) > 0
```

Бюджет/пополнение считается настроенным, если:

```js
Boolean(state.budget_rule_active) && Number(state.budget_limit) > 0
```

Если лимит расхода не настроен:

```text
Проблема лимит расхода = не установлен
```

Иначе:

```text
—
```

Если бюджет/пополнение не настроено:

```text
Проблема пополнение бюджета = не настроено
```

Иначе:

```text
—
```

Строка попадает в отчет, если хотя бы одно из двух полей проблемное.

## Отчет: Проблемы: вылеты лимитов

Отчет `Проблемы: вылеты лимитов` показывает РК, которые в выбранном периоде были неактивны из-за статуса `лимит расходов` или `нехватка бюджета` подряд не меньше 4 часов.

Источник на странице:

```text
drr-analytics -> limitActivity
```

Ключевая логика в локальном проекте:

```text
xway_dashboard_site/src/routes/drr-analytics-page.tsx
xway_dashboard_site/functions/_lib/catalog-issues.js
```

Константы дашборда:

```js
LIMIT_ACTIVITY_THRESHOLD_HOURS = 4
LIMIT_ACTIVITY_BATCH_SIZE = 6
LIMIT_ACTIVITY_DEADLINE_MS = 12000
```

Для локального полного сбора можно использовать больший deadline, но функция `collectCatalogIssues` все равно ограничивает один проход максимум 25 секундами. Поэтому сбор нужно запускать в цикле по `next_cursor`, пока `complete = true`.

### Название и файл

Markdown-заголовок:

```md

```

Безопасное имя файла:

```text

```

### Колонки отчета

Базовые колонки должны повторять отчет `Проблемы: настройка лимитов и бюджетов`, затем добавляются колонки времени неактивности и инцидентов:

```text
Артикул / XWAY / Название товара / Категория / Маркетолог / Остаток FBO / Тип РК / Проблема лимит расхода / Инциденты лимит расхода / Проблема пополнение бюджета / Инциденты пополнение бюджета / Макс. неактивность подряд / Всего неактивность / Инциденты всего
```

Значения проблем:

```text
Проблема лимит расхода = лимит расходов
Проблема пополнение бюджета = нехватка бюджета
```

Если по строке нет соответствующего типа проблемы, ставить:

```text
—
```

### Методы XWAY для вылетов

Сначала собрать тот же компактный каталог:

```js
const catalog = await collectCatalog(env, {
  start,
  end,
  mode: "compact",
  includeAux: false,
  forceRefresh: true,
})
```

Отобрать товары-кандидаты:

```text
campaign_state.status_code = ACTIVE или PAUSED
stock > 10
```

Для проверки вылетов использовать `collectCatalogIssues` со scope `limit_activity`:

```js
const result = await collectCatalogIssues(env, {
  productRefs,
  start,
  end,
  forceRefresh: true,
  cursor,
  deadlineMs: 25000,
  scope: "limit_activity",
})
```

Внутри `scope: "limit_activity"` используются:

```http
GET  https://am.xway.ru/api/adv/shop/{shop_id}/product/{product_id}/stata?is_active=0&start={start}&end={end}&tags&active_camps=1
POST https://am.xway.ru/api/adv/shop/{shop_id}/product/{product_id}/campaign/{campaign_id}/status-pause-history
```

Для `status-pause-history` тело запроса:

```json
{ "limit": 60 }
```

Если по конкретной РК ответ вернул полную страницу на 60 событий, считать историю потенциально обрезанной и точечно перезапросить эту РК:

```json
{ "limit": 120 }
```

В отчете в шапке фиксировать:

```text
Метод истории статусов: status-pause-history limit=60, fallback до 120 при полной странице
Fallback до 120: N
```

В этом режиме не нужно тянуть расписание РК, дневную exact-статистику и product daily stats. Проверяется только история статусов ACTIVE/PAUSED кампаний.

### Правила отбора вылетов

Строка попадает в отчет, если выполнены все условия:

1. Товар есть в XWAY.
2. `stock > 10`.
3. У товара есть РК со статусом `ACTIVE` или `PAUSED`.
4. РК не `FROZEN`.
5. В `status-pause-history` есть причина `limit` или `budget`.
6. По конкретной РК `max_incident_hours >= 4`.

Причины из истории статусов:

```text
limit  -> лимит расходов
budget -> нехватка бюджета
```

Если одна РК за период ловила обе причины, в одной строке заполняются обе проблемные колонки.

### Гранулярность строк

Строка отчета агрегируется на уровне:

```text
товар + тип РК
```

Если у одного товара несколько кампаний одного типа, их часы и инциденты суммируются в одну строку. Это сохраняет формат отчета `Проблемы: настройка лимитов и бюджетов`, где строка тоже привязана к товару и типу РК, а не к внутреннему ID кампании. Статус РК на момент сбора в `Тип РК` не выводить.

Тип РК выводить только в бизнес-терминах:

```text
Единая ставка
Ручная ставка: поиск
Ручная ставка: рекомендации
Ручная ставка: поиск + рекомендации
Оплата за клики
```

### Расчет колонок времени и инцидентов

Для каждой строки:

```text
Макс. неактивность подряд = максимум campaign.max_incident_hours среди вошедших кампаний
Всего неактивность        = сумма campaign.hours среди вошедших кампаний
Инциденты лимит расхода   = сумма campaign.incidents по issue.kind = limit
Инциденты пополнение бюджета = сумма campaign.incidents по issue.kind = budget
Инциденты всего           = сумма campaign.incidents среди вошедших кампаний
```

Формат часов:

```text
24 ч
13,6 ч
4,7 ч
```

Часы округлять до одного знака после запятой.

### Сортировка вылетов

Рекомендуемая сортировка:

1. `Макс. неактивность подряд` по убыванию.
2. `Всего неактивность` по убыванию.
3. `Остаток FBO` по убыванию.
4. Артикул и тип РК по алфавиту.

## Отчет: Проблемы: Автоисключения Поиска

Источник на странице:

```text
drr-analytics -> autoExclusions
```

Ключевая логика в локальном проекте:

```text
xway_dashboard_site/src/routes/drr-analytics-page.tsx
xway_dashboard_site/functions/_lib/catalog-auto-exclusions.js
```

### Название и файл

Markdown-заголовок:

```md
# Проблемы: Автоисключения Поиска
```

Имя файла:

```text
3. Проблемы: Автоисключения Поиска.md
```

### Колонки отчета

```text
Артикул / XWAY / Название товара / Категория / Маркетолог / Остаток FBO / Тип РК / Проблема автоисключения поиска / Правило автоисключения / Кластеры с тратами за 3 дня / Зафиксированные кластеры / Расход за период / Заказы всего
```

Колонка `XWAY` должна быть Markdown-ссылкой на настройку товара в XWAY:

```md
[XWAY](https://am.xway.ru/wb/shop/{shop_id}/product/{product_id})
```

Колонка `Кластеры с тратами за 3 дня` считается по проблемной РК из `campaignNormqueryStats`: считать `normquery`, если `expense > 0` за период отчета и `excluded != true`. То есть исключенные кластеры вычитаются из количества кластеров с тратами.

Колонка `Зафиксированные кластеры` считается по той же проблемной РК из `campaignNormqueryStats`: считать `normquery`, если `fixed = true`.

### Стартовый фильтр

Для этого отчета важно сразу сузить список артикулов, а не отправлять в детальную проверку все товары с любыми РК.

На старте брать только товары, где выполнены условия:

1. Товар есть в XWAY.
2. `stock > 10`.
3. В `campaign_states` есть хотя бы одна РК со статусом `ACTIVE` или `PAUSED`.
4. Ключ РК не равен `cpc`.

Исключать до детальной проверки:

```text
FROZEN
CPC / Оплата за клики
```

Перед записью строк в итоговый Markdown дополнительно исключать строки, где:

```text
Расход за период = 0
Тип РК = Единая ставка и Кластеры с тратами за 3 дня = 0
```

В отчете в шапке фиксировать:

```text
Фильтр строк: Расход за период > 0; для Единой ставки Кластеры с тратами за 3 дня > 0
```

В проекте XWAY это отражено отдельной функцией:

```ts
function isAutoExclusionCheckCampaignState(state: CatalogCampaignState) {
  if (!isLimitCheckCampaignState(state)) {
    return false;
  }
  return String(state.key || "").toLowerCase() !== "cpc";
}
```

Backend `collectCatalogAutoExclusions` все равно повторно отбрасывает `FROZEN` и `CPC` по детальному `productStata`. Это вторая линия защиты для смешанных товаров, где у одного артикула есть и CPM, и CPC.

### Методы XWAY для автоисключений

Сначала собрать компактный каталог:

```js
const catalog = await collectCatalog(env, {
  start,
  end,
  mode: "compact",
  includeAux: false,
  forceRefresh: true,
})
```

Потом проверить автоисключения:

```js
const result = await collectCatalogAutoExclusions(env, {
  productRefs,

```


### Правила проблем

РК считается проблемной, если `campaign.configured !== true`.

Если `Расход за период = 0`, строку не выводить в итоговый Markdown, даже если автоисключение не настроено.

Если `Тип РК = Единая ставка` и после вычитания исключенных кластеров `Кластеры с тратами за 3 дня = 0`, строку не выводить в итоговый Markdown. Для ручной ставки это правило не применяется.

Расшифровка:

```text
выключено     = правило не найдено, правило выключено или есть ошибка чтения правила
нет условий   = режим условий включен, но условия не заданы
не настроено  = прочий случай, когда configured=false
```

РК считается настроенной, если:

```text
rule.active = true
и fixed = false
```

или:

```text
rule.active = true
fixed = true
есть хотя бы одно условие автоисключения
```

## Формат сборного сообщения

Вместе с отчетными Markdown-файлами и инструкцией нужно присылать пользователю короткое сборное сообщение по маркетологам.

Источники для подсчета:

```text
1. Проблемы: настройка лимитов и бюджетов.md
2. Проблемы: вылеты лимитов.md
3. Проблемы: Автоисключения Поиска.md
```

Правила:

1. Группировать по колонке `Маркетолог`.
2. Количество проблем считать как количество строк в соответствующем Markdown-отчете.
3. Если у маркетолога по какому-то типу проблем `0`, этот тип не выводить.
4. Не писать слово `проблема:`.
5. Между названием типа проблемы и числом использовать `: `.
6. Если маркетолог не найден, использовать группу `—`.

Формат:

```text
маркетолог
- настройка лимитов и бюджет: N
- вылеты лимитов: N
- автоисключения поиска: N
```

Пример для текущей выгрузки:

```text
@a.beaver
- настройка лимитов и бюджет: 31
- вылеты лимитов: 6
- автоисключения поиска: 29

@a.manokhin
- настройка лимитов и бюджет: 3
- вылеты лимитов: 11
- автоисключения поиска: 3

@a.nekrasov
- вылеты лимитов: 17
- автоисключения поиска: 12

—
- настройка лимитов и бюджет: 2
- вылеты лимитов: 1
- автоисключения поиска: 2
```

## Важные нюансы

- Не использовать MPVibe.
- Не использовать XWAY product tags как маркетологов.
- Не использовать `FROZEN` РК.
- `PAUSED` нужно проверять вместе с `ACTIVE`.
- Если XWAY Cloudflare endpoint падает `1102`, запускать локальный сбор через прямые XWAY методы.
- Если Google Sheets API connector ловит `429`, использовать CSV-export.
- Тип РК должен отражать бизнес-термины, а не технические подписи `CPM`.
